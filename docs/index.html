01The Manifesto

Your skills are leaving your hands. The question is whether they'll still be yours when they get there.

For most of human history a skill lived in a person. You knew how to research, write, code, test, review work, and recover when something broke. You carried the standards, the shortcuts, the taste, and the checks that kept bad work from becoming real damage. AI is turning that into software. Your way of working can now become prompts, skill.md files, runbooks, scripts, MCP configs, permission boundaries, and agent workflows.

The thing that used to live in your hands can now live in a harness. That is one of the biggest opportunities in AI. It is also one of the biggest ownership fights. Because if your skills are going to live outside your hands, they should not belong to a rent-a-brain.

🔓

Not to Claude

Because it shipped the best skill format this quarter.

🔧

Not to Codex

Because it gave you the best workbench.

📝

Not to Cursor

Because the project started there.

💬

Not to ChatGPT

Because the subscription was convenient.

They should be yours. Visible, movable, inspectable, testable, and available wherever you work.

The gap

Today almost none of that is true. The prompt copies over. The intention copies over. The skill does not. So you rebuild it from memory, a teammate improves a different copy and never tells you, and your best workflow ends up stranded in a chat history nobody can find again.

You are not short on intelligence or context. You are short on a way to carry the procedure itself.

02The Scoping & Directories

Predictable file hierarchies binding skills globally or locally to your repository.

Global Scope

Lives in your user configuration directory at ~/.config/open-skills/. These skills are bound to your user session across all projects.

~/.config/open-skills/ ├── skills/ │ ├── personal-voice/ │ │ ├── skill.md │ │ └── voice-samples/ │ └── current-search/ │ └── skill.md └── runbooks/ └── content-pipeline.md

Local Scope

Lives inside your active project repository at .open-skills/. These skills are checked into git and shared across teammates working on the project.

your-project-repo/.open-skills/ ├── skills/ │ ├── browser-qa/ │ │ └── skill.md │ └── db-migration/ │ └── skill.md └── runbooks/ └── release-day.md

03Anatomy of a skill.md File

A skill is a markdown file parsed by code. It uses a strict YAML frontmatter block to define boundaries, required tools, and triggers.

skill.md
---
name: browser-qa
description: Verifies frontend changes across responsive breakpoints.
triggers:
  - "on UI change"
  - "before calling a layout task done"
boundaries:
  - "Do not test on production databases."
  - "Never bypass console error checks."
required_tools:
  - headless-browser
  - terminal
output_format: "QA_EVIDENCE.md"
---

## Objective
Ensure the active route renders correctly on desktop and mobile without runtime errors.

## Procedure
1. Identify the local server port. If not running, start it using `npm run dev`.
2. Open the browser tool and navigate to the modified routes.
3. Change viewport to 375x812 (Mobile) and inspect for overlapping text or broken layouts.

## Verification Contract (NON-NEGOTIABLE)
Your job is NOT done until you provide:
- [ ] Console logs showing zero uncaught exceptions.
- [ ] Saved screenshots of both Desktop and Mobile views in `./qa-artifacts/`.

Required Sections

  • YAML Frontmatter — Slugified name, description, trigger keywords, strict boundaries, required tools, and output format.
  • ## Objective — High-level description of what the procedure accomplishes.
  • ## Procedure — Step-by-step instructions.
  • ## Verification Contract (NON-NEGOTIABLE) — Markdown checklist that must be filled before completion.

04The Work Package Checklist

Eight checks that separate a skill you actually own from a lucky setup you happened to land in one app.

  1. Visible Can you find it without searching chat history?
  2. Movable Can you install it on a different platform in under 5 minutes?
  3. Inspectable Can you read the full procedure without running anything?
  4. Testable Does it have a verification step that confirms it worked?
  5. Versioned Does it have a version number and a changelog?
  6. Deps-declared Are all tool/data/API dependencies listed?
  7. Perm-bounded Are its permissions (network, files, keys) explicit?
  8. Platform-agnostic Does the core procedure work without platform-specific syntax?

05The One-Question Test

If I deleted this app tomorrow, how long would it take me to rebuild this workflow somewhere else?

07Runbooks as Composition

Runbooks chain primitive skills together. Instead of code, a runbook acts as a structural state-machine manifesto that instructs the agent how to pipe outputs from one skill to the next.

PhaseActive SkillInputExpected Verified Output
01media-transcriberRaw audio fileVerified .txt clean transcript
02personal-voiceClean transcriptMarkdown draft matching structural voice standards
03browser-qaLocal deployment routeScreenshot artifacts + zero console errors

# Start execution of a runbook phase list

08The Model Context Protocol (MCP) Server

Expose your skills and runbooks dynamically to Cursor, Claude Code, and Codex.

A local background process reads your local and global directories and exposes them to any active agent session. Exposes the following capabilities:

  • Prompts (open-skills-context) — Feeds the active procedures and structures directly into the system prompt context on agent boot up.
  • Tools (check_triggers) — Monitores the user's queries. If the query matches the triggers defined in the skill (e.g. "landing page changes"), it suggests or injects the appropriate skill.
  • Resources (open-skills://skills/local/{name}) — Exposes the raw markdown files directly as inspectable resources.
Launch MCP Server
$ python3 openskills.py mcp start
Starting Open Skills MCP Server...

09The Flywheel: session-to-skill Extractor

Save brilliant workflows from disappearing in chat logs.

The extractor reads the latest chat session logs from your local tools (e.g. Claude Code's log files or Superpowers SQLite databases), builds a transcript, and passes it to a local/completions model to automatically extract repeatable technical procedures as Open Skills.

$ python3 openskills.py extract --last-session

Retrieved session log (length: 59026 characters).
