01The Manifesto

Your skills are leaving your hands. The question is whether they'll still be yours when they get there.

For most of human history a skill lived in a person. You knew how to research, write, code, test, review work, and recover when something broke. You carried the standards, the shortcuts, the taste, and the checks that kept bad work from becoming real damage. AI is turning that into software. Your way of working can now become prompts, skill.md files, runbooks, scripts, MCP configs, permission boundaries, and agent workflows.

The thing that used to live in your hands can now live in a harness. That is one of the biggest opportunities in AI. It is also one of the biggest ownership fights. Because if your skills are going to live outside your hands, they should not belong to a rent-a-brain.

🔓

Not to Claude

Because it shipped the best skill format this quarter.

🔧

Not to Codex

Because it gave you the best workbench.

📝

Not to Cursor

Because the project started there.

💬

Not to ChatGPT

Because the subscription was convenient.

They should be yours. Visible, movable, inspectable, testable, and available wherever you work.

The gap

Today almost none of that is true. The prompt copies over. The intention copies over. The skill does not. So you rebuild it from memory, a teammate improves a different copy and never tells you, and your best workflow ends up stranded in a chat history nobody can find again.

You are not short on intelligence or context. You are short on a way to carry the procedure itself.

03Scoping & Directories

Predictable file hierarchies binding skills globally or locally to your repository.

Global Scope

Lives in your user configuration directory at ~/.config/open-skills/. These skills are bound to your user session across all projects.

~/.config/open-skills/ ├── skills/ │ ├── personal-voice/ │ │ ├── skill.md │ │ ├── tests/ │ │ │ └── test_basic.py │ │ └── adapters/ │ │ ├── hermes.md │ │ └── claude-code.md │ └── current-search/ │ └── skill.md └── runbooks/ └── content-pipeline.md

Local Scope

Lives inside your active project repository at .open-skills/. These skills are checked into git and shared across teammates working on the project.

your-project-repo/.open-skills/ ├── skills/ │ ├── browser-qa/ │ │ └── skill.md │ └── db-migration/ │ └── skill.md └── runbooks/ └── deploy-pipeline.md
Shadowing

When a local skill has the same name as a global skill, the local version takes precedence. The Web UI flags shadowed skills so you always know which version is active.

04Anatomy of a skill.md File

A skill is a markdown file parsed by code. It uses a strict YAML frontmatter block to define boundaries, required tools, and triggers.

skill.md
---
name: browser-qa
description: Verifies frontend changes across responsive breakpoints.
version: 1.0.0
triggers:
  - "on UI change"
  - "before calling a layout task done"
boundaries:
  - "Do not test on production databases."
  - "Never bypass console error checks."
required_tools:
  - headless-browser
  - terminal
output_format: "QA_EVIDENCE.md"
disable-model-invocation: false
user-invocable: true
---

## Objective
Ensure the active route renders correctly on desktop and mobile without runtime errors.

## Procedure
1. Identify the local server port. If not running, start it using `npm run dev`.
2. Open the browser tool and navigate to the modified routes.
3. Change viewport to 375x812 (Mobile) and inspect for overlapping text or broken layouts.

## Pitfalls
- Do not rely on cached screenshots — always capture fresh.
- Mobile Safari and Chrome render differently; test both if possible.

## Verification Contract (NON-NEGOTIABLE)
Your job is NOT done until you provide:
- [ ] Console logs showing zero uncaught exceptions.
- [ ] Saved screenshots of both Desktop and Mobile views in `./qa-artifacts/`.

Required Frontmatter Fields

FieldTypeDescription
namestringSlugified skill name (must match directory name)
descriptionstringOne-line description of the skill
versionstringSemantic version (e.g. 1.0.0)
triggerslistKeywords or phrases that activate the skill
boundarieslistHard constraints the skill must never violate
required_toolslistTools/APIs the skill depends on
output_formatstringExpected output filename or format

Required Sections

  • ## Objective — High-level description of what the procedure accomplishes.
  • ## Procedure — Step-by-step instructions an agent can follow.
  • ## Verification Contract (NON-NEGOTIABLE) — Markdown checklist that must be filled before completion.

Optional Sections

  • ## Pitfalls — Common failure modes and how to avoid them.

Package Structure

my-skill/ ├── skill.md # The procedure (required) ├── tests/ # Automated checks │ └── test_basic.py ├── adapters/ # Platform wrappers (auto-generated) │ ├── hermes.md │ ├── claude-code.md │ ├── cursor.md │ ├── codex.md │ └── generic.md ├── scripts/ # Helper scripts (optional) ├── templates/ # File templates (optional) └── references/ # Reference materials (optional)

05The Work Package Checklist

Eight checks that separate a skill you actually own from a lucky setup you happened to land in one app.

  1. Visible Can you find it without searching chat history?
  2. Movable Can you install it on a different platform in under 5 minutes?
  3. Inspectable Can you read the full procedure without running anything?
  4. Testable Does it have a verification step that confirms it worked?
  5. Versioned Does it have a version number and a changelog?
  6. Deps-declared Are all tool/data/API dependencies listed?
  7. Perm-bounded Are its permissions (network, files, keys) explicit?
  8. Platform-agnostic Does the core procedure work without platform-specific syntax?

06The One-Question Test

If I deleted this app tomorrow, how long would it take me to rebuild this workflow somewhere else?

07bSemantic Skill Recommendation

Rank skills by relevance to a task using a two-stage pipeline: deterministic pre-filter then LLM-backed semantic ranking.

How It Works

  1. Deterministic pre-filter — Trigger matching + keyword overlap on skill name and description. Produces a shortlist of up to RECOMMEND_CANDIDATE_CAP (default 20) candidates.
  2. LLM ranking — Sends compact metadata (name, description, triggers) to an LLM which returns a JSON array of ranked results with scores (0.0–1.0) and short reasons.
Degraded Mode

If no OPENROUTER_API_KEY is available, or the LLM call fails or times out, the recommendation falls back to the deterministic pre-filter results with llm_used: false. Hallucinated skill names from the LLM are silently dropped.

Access Surfaces

💻

CLI

openskills recommend "your task"

🔌

MCP Tool

recommend_skills(query="...", limit=5)

🌐

REST API

POST /api/skills/recommend

Configuration

VariableDefaultDescription
OPENROUTER_API_KEY—Required for LLM ranking. Falls back to degraded mode if missing.
RECOMMEND_MODELdeepseek/deepseek-v4-flashModel for recommendation ranking.
RECOMMEND_API_BASEhttps://openrouter.ai/api/v1/chat/completionsCompletions endpoint for recommendation.
RECOMMEND_CANDIDATE_CAP20Pre-filter shortlist size.
RECOMMEND_TIMEOUT_MS2000LLM timeout for recommendation.
OPENSKILLS_CONTEXT_MODEindexContext injection mode: index, full, or directive_only.

Context Injection Modes

ModeBehavior
indexInjects a directive to use recommend_skills plus a compact one-line-per-skill index. Default.
fullInjects full skill details (legacy behavior). All frontmatter and body text.
directive_onlyInjects only the directive to use recommend_skills. No skill listing.

Re-running connect updates the entry without disturbing other servers. A timestamped .bak backup is created before the first write to any file. Invalid writes are detected and rolled back from backup.

09The Model Context Protocol (MCP) Server

Expose your skills and runbooks dynamically to Cursor, Claude Code, and Codex.

A local background process reads your local and global directories and exposes them to any active agent session. Exposes the following capabilities:

  • Prompts (open-skills-context) — Injects skill context into the agent system prompt. Supports OPENSKILLS_CONTEXT_MODE=index|full|directive_only (default: index — directive + compact skill index).
  • Tools — recommend_skills, check_triggers, list_skills, get_skill, get_runbook_state, advance_runbook for programmatic access.
  • Resources — open-skills://skills/local/{name} and open-skills://skills/global/{name} expose the raw markdown files as inspectable resources.
Launch MCP Server
$ openskills mcp start
Starting Open Skills MCP Server...
Registered 5 tools, 2 resource templates, 1 prompt
Trigger Monitoring

The check_triggers tool scans user queries for keyword overlap against skill triggers to automatically suggest injecting relevant skills into the agent context.

Semantic Recommendation

The recommend_skills tool ranks skills by relevance to a task query using LLM-backed semantic matching with a deterministic fallback. Call it before starting a task instead of listing all skills.

# Start the backend (serves API + built SPA)

INFO:     Uvicorn running on http://0.0.0.0:8001

# Or for development (hot-reload)
$ cd web && npm run dev       # Vite dev server on :5174

Authentication

By default, the server runs without authentication (suitable for localhost). To enable shared-secret auth, set OPENSKILLS_API_TOKEN:

$ export OPENSKILLS_API_TOKEN=your-secret-token
$ python3 server.py

When enabled, all API requests must include an Authorization: Bearer <token> header.

11The Flywheel: Session-to-Skill Extractor

Save brilliant workflows from disappearing in chat logs.

The extractor reads the latest chat session logs from your local tools (Claude Code history logs or Superpowers SQLite databases), builds a transcript, and passes it to DeepSeek V4 Flash via OpenRouter to automatically extract repeatable technical procedures as Open Skills.

$ openskills extract --last-session
Retrieving latest session chat logs...
Retrieved session log (length: 59026 characters).
Calling OpenRouter API to extract repeatable procedures...
✓ Extracted skill 'ai-chat-frontend-design' and wrote draft to:
  .open-skills/skills/pending-review/ai-chat-frontend-design/skill.md
  You can now review, modify, and commit it.

Configuration

VariableDefaultDescription

EXTRACT_MODELdeepseek/deepseek-v4-flashModel to use for extraction.
EXTRACT_API_BASEhttps://openrouter.ai/api/v1/chat/completionsOpenRouter completions endpoint.

LLM-Assisted Fix Suggestions

When a skill fails validation, the Web UI can request an LLM-suggested fix via the POST /api/skills/{name}/suggest-fix endpoint. The fix is generated by the same DeepSeek model and returned as a corrected skill body you can review and apply.
